# Spec: workflow-runtime

## Purpose

Контракт долговременного выполнения воркфлоу бронирования на Temporal: корректность регистрации активностей, retry/timeout-политики, ограничение Event History через Continue-As-New, конфигурируемые таймауты ожидания и версионирование изменений логики. Эта capability описывает *как* воркфлоу исполняется долговременно и устойчиво, в отличие от `hotel-negotiation`, описывающей *что* воркфлоу делает (логику переговоров).

## Requirements

### Requirement: Корректная регистрация активностей воркера
Воркер Temporal SHALL регистрировать активности как связанные `@activity.defn`-методы объекта `ConciergeActivities`, а не как сам инстанс. Конструкция `Worker(...)` MUST завершаться без ошибок при наличии реального Temporal-сервера.

#### Scenario: Воркер стартует против реального Temporal
- **WHEN** воркер собирается через `build_worker(client, settings, activities)` и запускается против настроенного `temporal_target`
- **THEN** `Worker(...)` конструируется без `TypeError` и начинает опрашивать task queue

#### Scenario: Все активности зарегистрированы по имени
- **WHEN** воркер зарегистрирован
- **THEN** методы `extract`, `discover_contact`, `agent_turn`, `send_email`, `build_report`, `relay_to_client`, `update_booking_state`, `record_inbound_reply` доступны для вызова из воркфлоу по своим именам

#### Scenario: E2E-тест использует ту же регистрацию
- **WHEN** выполняется интеграционный тест под `KKR_E2E_TEMPORAL=1`
- **THEN** воркфлоу запускается через `env.new_worker(...)` с корректным списком bound-методов и не падает на регистрации

### Requirement: Ограниченные retry-политики для активностей
Активности MUST запускаться с явной ограниченной `RetryPolicy`. LLM-активности (`extract`, `agent_turn`) SHALL иметь не более 3 попыток; сайд-эффектные идемпотентные активности (`send_email`, `relay_to_client`, `update_booking_state`, `build_report`) — не более 5; `discover_contact` — не более 3. Ни одна активность MUST NOT использовать дефолтную политику Temporal (бесконечный retry).

#### Scenario: LLM-активность не уходит в бесконечный retry
- **WHEN** активность `agent_turn` стабильно падает (провайдер LLM недоступен)
- **THEN** после исчерпания числа попыток активность завершается ошибкой, а не повторяется бесконечно

#### Scenario: Идемпотентная сайд-эффектная активность ретраится ограниченно
- **WHEN** активность `send_email` падает из-за временной недоступности шлюза
- **THEN** она повторяется не более 5 раз с экспоненциальным backoff, после чего воркфлоу получает ошибку активности

### Requirement: Явные таймауты активности
LLM-активности SHALL иметь `start_to_close_timeout` не менее 180 секунд; прочие активности — 120 секунд. Таймауты MUST быть определены явно (не дефолтные).

#### Scenario: LLM-активность имеет расширенный таймаут
- **WHEN** вызывается `extract` или `agent_turn`
- **THEN** их `start_to_close_timeout` составляет не менее 180 секунд

### Requirement: Ограничение Event History через Continue-As-New
Воркфлоу MUST периодически сбрасывать Event History через `workflow.continue_as_new`, когда число завершённых «туров переговоров» в текущем run достигает `workflow_continue_as_new_threshold`. Порог MUST быть настраиваемым (по умолчанию 5). Состояние `BookingState`, текущий триггер переговоров и необслуженные элементы очередей сигналов SHALL передаваться в новый run (через `RunInput(resume=...)`), чтобы выполнение продолжалось с того же места.

#### Scenario: Сброс истории после порога туров
- **WHEN** воркфлоу завершил число туров переговоров, равное порогу continue-as-new
- **THEN** инициируется новый run с чистой историей, которому передано текущее состояние и триггер

#### Scenario: Сигнал «в полёте» не теряется при сбросе
- **WHEN** во время сброса истории в очереди сигналов есть необслуженный ответ/фоллоуап
- **THEN** новый run получает этот сигнал и обрабатывает его

#### Scenario: Воркфлоу сохраняет long-lived-семантику
- **WHEN** воркфлоу долгоживущий (превышает порог continue-as-new)
- **THEN** его Event History остаётся ограниченной и не достигает серверного лимита

### Requirement: Конфигурируемые таймауты ожидания
Таймауты ожидания ответа отеля, уточнения от клиента и реактивации SHALL передаваться как параметры `run()` (с дефолтами из `Settings`), а не быть захардкоженными в теле воркфлоу.

#### Scenario: Таймаут реактивации задаётся параметром
- **WHEN** `TemporalWorkflowGateway` стартует воркфлоу
- **THEN** таймаут реактивации передаётся как аргумент `run()` из `Settings`, а не из литерала в теле воркфлоу

### Requirement: Контракт версионирования изменений логики воркфлоу
Любое изменение структуры команд воркфлоу (добавление/удаление активности, изменение порядка оркестрации, новые ветви в `_negotiate`) MUST быть обёрнуто в маркер `workflow.patched("<change-id>")` с уникальным идентификатором изменения, чтобы не ломать replay уже запущенных long-lived воркфлоу.

#### Scenario: Изменение логики помечается патчем
- **WHEN** разработчик изменяет порядок или состав команд в `run()`/`_negotiate()`
- **THEN** изменение вводится через `workflow.patched("<change-id>")` так, что уже запущенные воркфлоу проходят replay по старой логике, а новые — по новой
