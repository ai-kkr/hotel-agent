# Spec Delta: booking-intake

## MODIFIED Requirements

### Requirement: Ослабленная sender-аутентификация для chat-origin клиентов
Для клиентов, чья идентичность установлена через `ChannelSession` (chat-origin), система SHALL
аутентифицировать intake по сессии канала, а не по SPF/DKIM отправителя. Персональный ящик
`c.<token>@kkr-hotel.com` chat-origin клиента SHALL раскрываться самому клиенту (например, в
приветствии на `/start`) и SHALL служить индивидуальным адресом для пересылки подтверждения по email;
в этом случае система SHALL аутентифицировать такое письмо по факту владения токеном в адресе
(capability), так как токен высокоэнтропийный и индивидуален для клиента. Email-channel клиенты
(имеющие зарегистрированный email) SHALL по-прежнему проходить строгую SPF/DKIM-проверку совпадения
отправителя без изменений.

#### Scenario: Chat-origin intake через чат-сессию
- **WHEN** chat-origin клиент инициирует intake через чат-сессию
- **THEN** intake выполняется от имени этого клиента без требования совпадения email-отправителя

#### Scenario: Chat-origin forward по раскрытому адресу принят по владению токеном
- **WHEN** chat-origin клиент (без зарегистрированного email) пересылает подтверждение на свой
  раскрытый `c.<token>@`
- **THEN** система принимает письмо по факту владения токеном в local-part и выполняет intake от
  имени этого клиента

#### Scenario: Email-channel клиент — строгая проверка сохранена
- **WHEN** email-channel клиент (с зарегистрированным email) шлёт forward на `c.<token>@`
- **THEN** система по-прежнему требует и проверяет совпадение отправителя с зарегистрированным email
  клиента (без изменений)

#### Scenario: Неизвестный токен отклоняется
- **WHEN** входящее письмо адресовано на `c.<token>@` с неизвестным токеном
- **THEN** система отклоняет письмо независимо от канала происхождения клиента
