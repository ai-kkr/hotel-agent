## Context

Агент — один ReAct-граф `StateGraph` (`model` + `tools` + `cleanup`), исполняемый Temporal
LangGraph-плагином (каждая нода — активность). История переписки живёт в `state["messages"]`
(`EmailState` extends `AgentState`, штатный `add_messages`-редьюсер). Состояние персистится в
таблице `states` (`StateORM`, JSONB) через `load_state`/`save_state` активности — **без**
LangGraph-checkpointer'а. Через Temporal data converter (`message_aware_data_converter`)
ходят langchain-сообщения и `Command`.

Уже есть `cleanup`-компактизация: в конце хода тяжёлый вывод `search_internet`/`extract_web_page`
замещается стабом на месте (id-upsert). Она режет только одноразовый поиск — на рост самой
беседы не влияет. Когда `messages` расползается (длинный тред с отелем, много ходов с гостем),
вход модели упирается в контекстное окно. Нужен второй эшелон — LLM-саммаризация.

## Goals / Non-Goals

**Goals:**
- Не упасть и не деградировать на длинных переписках: при приближении к контекстному окну
  мягко сжимать старую часть в бегущее саммари.
- Сохранить связность: саммари фокусируется на текущей цели и **следующем шаге**, несёт вехи,
  решения и данные — агент продолжает без шва.
- Точный, бесплатный замер размера контекста — без локального токенайзера и новой зависимости.
- Не сломать existing-инварианты: сериализуемость стейта, Temporal-граница, tool_call↔ToolMessage
  связность, потоковая отправка ответов из активностей.

**Non-Goals:**
- Защита первого хода от гигантского пересланного `.eml` (нет прошлого `usage`).
- Жёсткий crash-барьер на overflow окна — это мягкий порог.
- Смена/удаление `cleanup`-компактизации.
- Ручной тул саммаризации.

## Decisions

### D1. Сигнал размера контекста — `usage_metadata` ответа модели, а не локальный токенайзер

**Решение:** `model_node` после `result = await model.ainvoke(...)` читает
`result.usage_metadata["input_tokens"]` (реальный размер `[SYSTEM_MAIN, *messages]`, как его
видит провайдер) и пишет в новое поле `last_prompt_tokens: int`. Проверка саммаризации читает
его при следующем входе в `model_node`.

**Почему:** точная цифра от самого провайдера, ноль новых зависимостей (tiktoken и т. п.), нет
рассинхрона «считаем локально — а модель режет иначе». Модель уже отдаёт usage в каждом ответе.

**Альтернативы:** (a) `tiktoken` локально — отвергнуто (зависимость + приближение, расходится с
реальным токенизатором openrouter/zai-моделей); (b) char-count эвристика — грубо, только как
fallback для первого хода (см. Non-goals).

**Следствие:** триггер по природе **реактивный** — порог проверяется по *прошлому* вызову.
Защита включается со второго model-вызова. Это сознательное ограничение.

### D2. Бегущее саммари — отдельное поле стейта `conversation_summary: str | None`, не сообщение

**Решение:** саммари хранится в `EmailState.conversation_summary` (чистая строка, сериализуется
в JSONB). `model_node` препендит его как `SystemMessage` при invoke:
`[SYSTEM_MAIN, *(SystemMessage(summary) if summary else []), *messages]`.

**Почему:**
- **Порядок:** `add_messages` дописывает новые сообщения в конец. Если бы саммари был сообщением,
  он оказывался бы *после* recency-окна. Отдельное поле — всегда первое.
- **Аккумуляция:** повторная саммаризация берёт `[prev_summary, *new_old_prefix]` — прежнее
  саммари не теряется, слои не наслаиваются.
- **Чистая история:** большая инструкция «как обобщать» — одноразовый invoke, не остаётся в
  `messages` навсегда.

**Альтернатива (буквально из исходного запроса):** вставить сообщение-инструкцию в историю,
модель выдаёт саммари как ход, затем удалить всё кроме саммари. Отвергнуто по трём пунктам выше
(плюс саммари-сообщение пришлось бы отдельно помечать, чтобы его не сжала `cleanup` / не
переиспользовал редьюсер некорректно).

### D3. Удаление старого префикса — `RemoveMessage`, recency-окно, гарантия целостности tool-call

**Решение:** `summarize_node` срезает префикс `messages[:-k]`, саммаризует его одним invoke
(вход: `[SUMMARIZE_PROMPT, *(SystemMessage(prev_summary) if prev else []), *prefix]`), пишет
результат в `conversation_summary`, возвращает
`[RemoveMessage(id=m.id) for m in prefix]`. Штатный `add_messages` выкидывает префикс по id;
recency-окно `messages[-k:]` остаётся нетронутым.

**Гарантия целостности (критично):** граница среза **не может** пройти между
`AIMessage(tool_calls)` и его `ToolMessage(s)`. Если эвристика «оставить последние k» режет
пару — откатываемся назад до безопасной границы (до `AIMessage` с tool_calls). Иначе модель
увидит `ToolMessage` без породившего вызова и сломается.

**Почему `RemoveMessage`, а не полный replace:** свой редьюсер или полный перезапись `messages`
значит ломать инвариант `add_messages` (на нём держится и id-upsert из `cleanup`). `RemoveMessage`
— штатный, идемпотентный, дружит с id-upsert.

**Размер окна** `summarize_keep_last_messages` (дефолт 6) — несёт последний ход гостя + его
работу тулов/ответ. Этого хватает, чтобы модель действовала дальше; остальное покрывает саммари.

### D4. Отдельная нода `summarize` + conditional `summarize_check`, не inline в `model_node`

**Решение:** граф становится
`START → summarize_check → summarize → model` (или `→ model`); `tools → summarize_check`
(вместо нынешнего `tools → model`). `summarize_check` — conditional, читает `last_prompt_tokens`,
возвращает `"summarize" | "model"`.

**Почему отдельная нода, несмотря на правило «предпочесть reuse графа»:** (a) чистая
activity-граница — у саммаризационного LLM-вызова свой `start_to_close_timeout`
(`llm_activity_timeout_seconds`, т.к. это полноценный LLM-вызов), своя retry-policy, свой
Langfuse trace-сегмент; (b) `model_node` остаётся с одной ответственностью (вызвать модель,
отправить ответ, записать usage); (c) консистентно с `cleanup` (там тоже отдельная нода ради
разделения забот). Inline-вариант сэкономил бы ноду, но смешал бы два LLM-вызова в одной
активности без чистой observability-границы.

**Альтернатива:** inline в начале `model_node`. Отвергнуто — теряется чистая activity/trace.

### D5. Промпт саммаризации — отдельный файл, одноразовый скрытый вызов (cache-aware)

**Решение:** `src/agent/prompts/system_summarize.md` — не ход агента, не тул. Один invoke с
тематическим фокусом. Содержание:
- **Фокус:** текущая цель + непосредственно следующий шаг (что агент делал/собирался сделать).
- **Сохранить обязательно:** зафиксированная бронь, найденный email отеля, отправленные письма
  (тема, суть, `message_id`), ответы отеля (содержание + решения), плановые задачи, пожелания
  гостя, открытые вопросы, принятые решения.
- **Сжимать болтовню, не факты.** Замечание: бронь/пояса/`message_id` и так живут в полях
  `EmailState` — саммари упоминает их для связности, но не должен реэкстрактировать как
  единственный источник.

**Формирование вызова (cache-critical — «префикс немутабелен»):** кеш длинного треда держится
только если саммаризационный вызов — это **продолжение** того же запроса, что кешировал `model_node`,
а не отдельный запрос. Три жёстких условия:

1. **Модель идентична, включая тулы.** `summarize_node` строит модель ровно как `model_node`:
   `build_model(settings).bind_tools(tools, **sticky_session_kwargs(model, client_id))` — тот же
   провайдер, те же привязанные тулы, тот же OpenRouter `session_id`. Любое отличие (тулы
   привязаны/нет, другой `session_id`) инвалидирует кеш с первого байта. (Функционально тулы
   саммаризации не нужны — они нужны для cache-parity.)
2. **Порядок сообщений зеркалит `model_node`.** Инпут:
   `[SYSTEM_MAIN, SystemMessage(prev_summary)?, *prefix, HumanMessage(SYSTEM_SUMMARIZE)]`.
   - `SYSTEM_MAIN` — системный промпт, первый (его НЕ заменяет промпт саммаризации);
   - `SystemMessage(prev_summary)` — бегущее саммари **в той же позиции, что и в `model_node`**
     (сразу после `SYSTEM_MAIN`, а не в конце). При ре-сжатии `prev_summary` в точности равен тому,
     что `model_node` уже отправлял, → `[SYSTEM_MAIN + summary + история]` побайтово совпадает с
     кешированным префиксом → cache-hit;
   - `*prefix` — сжимаемая старая история (префикс того, что отправлял `model_node`);
   - `HumanMessage(SYSTEM_SUMMARIZE)` — инструкция саммаризации, **единственная** не из кеша часть,
     последняя.
3. **Префикс немутабелен.** Ничто из кеша не переставляется и не модифицируется — только
   дописывается хвостовая `HumanMessage`.

Свап `SYSTEM_MAIN → SYSTEM_SUMMARIZE` (или убрать `bind_tools`) ломал бы весь кеш треда на каждой
саммаризации — на длинной переписке это дорого. Ранний вариант «положить `prev_summary` в хвостовую
`HumanMessage`» отвергнут по той же причине: саммари в конце ≠ саммари в начале у `model_node`, кеш
мимо.

### D6. Уведомление гостя перед саммаризацией

**Решение:** перед саммаризационным invoke — `send_telegram_reply("⏳ Переписка стала длинной —
подвожу итог, чтобы ничего не потерять. Это займёт немного времени.")`. Тот же путь, что у
`model_node`. Не тул, не ответ агента — push из ноды (по аналогии с typing-indicator).

### D7. Настройки

`summarize_token_threshold: int = 100000` (мягкий порог; должен быть ниже окна модели —
подгоняется под конкретный prod-model). `summarize_keep_last_messages: int = 6`. Оба в
`config.py` + `config.yaml`, префикс `KKR_`.

## Risks / Trade-offs

- **[Реактивный триггер не ловит первый ход]** → принятая цена за отказ от токенайзера.
  Документируем; при необходимости позже добавим char-эвристику только для первого хода.
- **[Срез пары tool_call↔ToolMessage]** → гард в `summarize_node`: откат к безопасной границе.
  Покрыть интеграционным тестом (сцена «модель вызвала тул, затем превентивная саммаризация»).
- **[Саммари потеряло факт]** → качеством управляет промпт (D5) + то, что ключевые факты
  дублированы в полях `EmailState` и переживают сжатие. На очень длинных тредах — присущий
  LLM-сжатию риск; компенсируем recency-окном.
- **[Двойной LLM-вызов на ходе саммаризации ( latency / cost)]** → случается редко (только при
  переходе через порог), уведомление гостя обосновывает задержку. Если саммаризация всё ещё
  оставляет `last_prompt_tokens > threshold` (патология) — повтор не зацикливается: после сжатия
  окно маленькое, следующий вход будет ниже порога.
- **[Два `SystemMessage` подряд (`SYSTEM_MAIN` + summary)]** → все поддерживаемые провайдеры
  (openai / zai / openrouter) это принимают; проверено существующим паттерном.
- **[Сериализация/Temporal-граница]** → новые поля — чистый JSON (`str | None`, `int`); new
  non-JSON field **не появляется**, data converter не трогается. `RemoveMessage` — это
  langchain-сообщение, уже round-trip-ится конвертером.

## Migration Plan

1. Реализация (см. tasks.md): state-поля → нода+conditional+edges → промпт → настройки →
   интеграционные тесты.
2. `uv run ruff check && uv run ruff format && uv run ty check`.
3. Локально: `docker compose up -d postgres temporal`, прогнать тесты, руками проверить длинный
   тред (довести до порога, убедиться что саммари сработало и агент продолжил осмысленно).
4. **Миграции БД нет** (новые поля в существующем JSONB). Коммит в `main`, деплой
   `railway up --service app --detach -m "…"`, поллинг `railway deployment list` до SUCCESS.
5. **Rollback:** revert-коммит + redeploy. Старые записи `states` с уже записанным
   `conversation_summary` просто игнорируются старой версией (extra-поля в JSONB
  ypyтяны не будут; pydantic `extra="ignore"` их выкинет) — совместимость обратно/вперёд
   сохраняется.
