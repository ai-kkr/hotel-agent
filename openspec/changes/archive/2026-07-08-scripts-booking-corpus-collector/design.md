## Context

Продуктивный сервис `kkr-hotel-assist` принимает подтверждения брони как **пересланные (forward)** письма на `c.<token>@kkr-hotel.com` и в `booking-intake` разделяет обложку клиента (пожелания) и пересланное содержимое, извлекает данные брони, валидирует отправителя. Для ручной отладки всей этой цепочки нужен корпус писем, которые выглядят именно так — как форвард от клиента, — но собранных из реальных подтверждений разных систем бронирования.

Владелец имеет в личном Gmail много реальных подтверждений от разных систем (Booking.com, Expedia, Marriott, независимые отели и т.д.). Задача — автоматически собрать N разнообразных и переупаковать в replay-ready форвард-`.eml`.

Ограничения проекта (из `openspec/project.md`): Python, Clean Architecture, зависимости только через `uv`/`pyproject.toml`, типы обязательны, тесты обязательны. Продуктивный код не должен зависеть от этой dev-утилиты.

## Goals / Non-Goals

**Goals:**
- Собрать из личного Gmail N (по умолчанию 10) разнообразных подтверждений брони отеля.
- Переупаковать каждое в форвард-вид, пригодный для ручного прогона intake-цепочки (тренирует форвард-парсинг, экстракцию данных брони и захват пожеланий, а не только экстрактор).
- Минимально-инвазивный вход в Gmail: без 2FA, без ручной выгрузки Takeout.
- Жить как отдельный dev-only пакет `scripts/`, никак не связанный с `src/`.

**Non-Goals:**
- Не является инструментом продакшена; не вызывается из приложения.
- Не делает анонимизацию/редакцию PII — данные личные и в репо не попадают.
- Не реализует автоматический replay в webhook (владелец прогоняет `.eml` вручную; возможная будущая утилита — вне скоупа).
- Не покрывает сценарий B (живой forward через реальный Mailgun-домен).
- Не использует Temporal/LangGraph-агента: поиск делает Gmail, классификация — один батч structured-extraction, цикл не нужен.

## Decisions

### D1. Доступ к Gmail — `ezgmail` (OAuth), без 2FA
**Решение:** `ezgmail` поверх Gmail API с OAuth Desktop-credential.

**Почему не альтернативы:**
- *IMAP + X-GM-RAW* — даёт UI-поиск и чистый `.eml`, но требует app-password, а он доступен только при включённой 2FA. Владелец не хочет включать 2FA. → отброшено.
- *Takeout `.mbox`* — ноль настройки и без 2FA, но ручная выгрузка всего ящика и офлайн-обработка. → отброшено в пользу живого поиска.
- *Browser automation (Playwright/Selenium)* — повторяет «логин в браузере», но скрейпинг DOM Gmail обфусцирован и хрупок, нет чистого `.eml`, риск антибот-челленджа. → отброшено.

**Сетап (one-time):** Cloud Console → Gmail API → OAuth Desktop client → `credentials.json`. Первый запуск открывает браузер, обычный логин по паролю (без 2FA), consent → сохраняется refresh-токен, далее браузер не нужен. `ezgmail.init()`/`search()` дают UI-поиск (`q=`) и объекты писем.

### D2. Форма «агента» — plain Python + один батч structured-extraction, без графа
**Решение:** поиск делегируется Gmail (`ezgmail.search(...)`); LLM участвует ровно одним батчем `model.with_structured_output(Schema)` для классификации «это подтверждение отеля? + какая система/отправитель». Цикла/графа нет.

**Почему:** после сужения поиска кандидатами до ~десятков итеративный агент избыточен. Альтернатива — `create_agent` с тулзами `scan`/`save`/`list` — была бы гибче, но дороже по токенам и менее предсказуема для разовой dev-тулзы. Решение не противоречит конвенциям проекта (прежнее жесткое правило `create_agent` удалено из памяти).

### D3. Разнообразие — потоковый first-seen-per-система с fallback на жадный добор
**Решение:** после классификации кандидаты группируются по системе/отправителю; берётся по одному от каждой системы, пока не наберётся N. Если уникальных систем меньше N — добираются оставшиеся кандидаты.

**Почему:** жадная максимизация разброса избыточна для разового сбора; потоковый дедуп проще и достаточно хорош. Альтернатива (LLM-кластеризация по «похожести формата») — оверкилл.

### D4. Слой A+B — скрипт переупаковывает оригинал в форвард
**Решение:** финальный шаг пересобирает каждое письмо:
- `From:` — email владельца (как зарегистрированный клиент; берётся из конфига/env).
- `To:` — `c.<token>@<mail-domain>` (параметризуется).
- `Subject:` — `Fwd: <оригинальный subject>`.
- Тело: опциональная обложка-пожелания (из небольшого пула шаблонов или пусто) + `-----Original Message-----` + оригинальные заголовки/тело.
- Сохраняется как `.eml` (стандартная сериализация `email.message`).

**Почему A+B, а не только A (оригинал как есть):** оригинал отвалятся на аутентификации/роутинге и не напряжёт форвард-парсинг — а именно он — самая ценная часть intake. Бонус: варьируя обложку-пожелания, заодно стресс-тестим захват пожеланий клиента.

### D5. Пакет `scripts/`, зависимости в `dev`-группе
**Решение:** новый каталог `scripts/` на корне репо (сиблинг `src/`); `ezgmail` добавляется в `[project.optional-dependencies].dev`. Продуктивный код (`src/...`) не импортирует из `scripts/`. Вывод (`.eml`, `manifest.json`, `credentials.json`, refresh-токен) — в папке вне репо / покрыт `.gitignore`.

### D6. LLM-вызов использует существующий стек проекта; тесты — с fake
**Решение:** structured-extraction через модель, инициализируемую по существующей конфигурации проекта. Тесты логики классификации/переупаковки гоняются без живого LLM (fake-модель по конвенции проекта), без сети и без живого Gmail (поиск мокается).

## Risks / Trade-offs

- **[PII владельца попадёт в git]** → папка вывода, `credentials.json` и токен добавляются в `.gitignore`; README явно предупреждает; анонимизация сознательно не делается (non-goal).
- **[OAuth app в статусе Testing → предупреждение «не верифицировано»]** → для личного использования приемлемо; владелец кликает «Advanced → Go to app». Документируем в README.
- **[Слишком узкий поисковый запрос → мало кандидатов / мало разнообразия]** → запрос параметризуется и расширяется; D3 fallback добирает оставшееся; логируется, сколько кандидатов найдено и сколько систем.
- **[LLM-классификация ошибается на граничных случаях]** → порог уверенности в Schema; неоднозначные кандидаты пропускаются, а не попадают в корпус; логируется причина пропуска.
- **[Несоответствие формата форварда тому, что ожидает intake]** → формат обложки + `-----Original Message-----` выравнивается с тем, как `booking-intake` разделяет блоки (проверяется по существующей спеке/парсеру на первом собранном письме).
- **[Google отзывает/протухает токен]** → при ошибке аутентификации скрипт падает с понятным сообщением и инструкцией повторить consent.
