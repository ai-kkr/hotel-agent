## Why

Чтобы вручную прогонять и отлаживать intake-цепочку (`booking-intake`) и дальнейшие переговоры на реалистичных данных, нужен корпус реальных писем-подтверждений брони отеля, переупакованных так, как будто клиент переслал их на `c.<token>@kkr-hotel.com`. Сейчас таких тестовых данных нет, а синтетические письма не напрягают форвард-парсинг и экстрактор данных брони.

## What Changes

- Добавляется отдельный dev-only пакет `scripts/` (сиблинг `src/`, не часть приложения) с утилитой сбора корпуса писем.
- Утилита подключается к личному Gmail владельца через `ezgmail` (OAuth, браузер-логин один раз, без 2FA), ищет подтверждения бронирования через нативные операторы поиска Gmail (`q=`).
- Один батч LLM-экстракции (`model.with_structured_output`) классифицирует кандидатов и определяет систему/отправителя для обеспечения разнообразия; явного агента/графа нет.
- Из кандидатов отбирается N (по умолчанию 10) разнообразных (по системе/отправителю) подтверждений.
- Каждое выбранное письмо переупаковывается в **форвард-вид** (слой A+B): `From:` — владелец (как зарегистрированный клиент), `To:` — `c.<token>@kkr-hotel.com`, тело — опциональная обложка-пожелания + `-----Original Message-----` + оригинальное подтверждение.
- Результат сохраняется как отдельные `.eml` файлы (+ опциональный `manifest.json` с метаданными) в локальную папку вне репозитория.
- Добавляются dev-зависимости (`ezgmail`, при необходимости — для LLM-вызова) в группу `[project.optional-dependencies] dev`.

## Capabilities

### New Capabilities
- `booking-corpus-collector`: dev-утилита для сбора N разнообразных реальных подтверждений брони отеля из личного Gmail и переупаковки их в replay-ready форвард-`.eml` для ручного прогона intake-цепочки.

## Impact

- **Код:** новый пакет `scripts/` (вне `src/`); продуктивный код (`src/domain`, `src/infrastructure`, `src/presentation`) не затрагивается.
- **Зависимости:** `pyproject.toml` — добавление `ezgmail` (и использование уже имеющегося LLM-стека проекта) в `dev`-группу.
- **Данные/приватность:** собираемые `.eml` содержат личные PII владельца и **никогда не попадают в репозиторий** (папка вывода вне репо / в `.gitignore`); анонимизация не требуется.
- **Тестирование:** продуктивный код не меняется, новых миграций/БД-изменений нет; покрывается тестами логика экстракции и переупаковки форварда (с LLM-fake по конвенции проекта).
