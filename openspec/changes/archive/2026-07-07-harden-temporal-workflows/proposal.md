## Why

Временной слой (Temporal) выстроен идиоматично, но не готов к длительной эксплуатации. Конкретно: (1) регистрация активностей падает при первом же реальном запуске воркера — передаётся инстанс вместо `@activity.defn`-методов; (2) все активности используют дефолтную политику Temporal (бесконечный retry) и единый `start_to_close_timeout=120s`, что для LLM-вызовов создаёт риск «вечного» 120-секундного цикла и неконтролируемых трат; (3) воркфлоу по дизайну живёт до 30 дней (D11, long-lived reactivation) без Continue-As-New, поэтому Event History растёт без ограничений; (4) нет стратегии версионирования `workflow.patcher` — любое изменение кода `run()` ломает replay уже запущенных воркфлоу. Сейчас это критично, потому что именно на этом слое держится весь pipeline бронирования.

## What Changes

- **BREAKING (исправление регистрации)**: воркер регистрирует связанные `@activity.defn`-методы, а не инстанс `ConciergeActivities`. Без этого `Worker(...)` падает с `TypeError: Activity <unknown> missing attributes` на старте.
- **Retry-политики**: явные `RetryPolicy` для активностей. LLM-активности (`extract`, `agent_turn`) — ограниченное число попыток; сайд-эффектные (`send_email`, `relay_to_client`, `update_booking_state`, `build_report`) — ограниченный retry с учётом их идемпотентности.
- **Таймауты**: пересмотр `start_to_close_timeout` для LLM-активностей; добавление `retry`-политик как первичного ограничителя.
- **Continue-As-New**: воркфлоу периодически сбрасывает историю, передавая `BookingState` в новый run; задаётся лимит «туров переговоров» на один run.
- **Версионирование**: введён `workflow.patcher`-маркер для будущих изменений логики `run()`/`_negotiate()`, чтобы не ломать replay уже запущенных long-lived воркфлоу.
- **Конфигурируемые таймауты**: захардкоженные в теле воркфлоу 14/30-дневные ожидания выносятся в параметры `run()` / настройки.
- **E2E**: тест `test_workflow_integration` и новый e2e-сценарий используют корректную регистрацию и покрывают retry/continue-as-new.

## Capabilities

### New Capabilities
- `workflow-runtime`: контракт долговременного выполнения воркфлоу на Temporal — корректность регистрации активностей, retry/timeout-политики, ограничение истории через Continue-As-New, версионирование изменений логики, идемпотентность старта.

## Impact

- **Код**: `src/infrastructure/workflows/worker.py` (регистрация), `workflow.py` (retry-политики, Continue-As-New, версионирование, параметры), `temporal_gateway.py` (параметры run / возможный `workflow_run_timeout`), `config.py` (новые настройки retry/continue-as-new).
- **Тесты**: правка регистрации в `tests/workflows/test_workflow_integration.py`; новые юнит/интеграционные сценарии под `KKR_E2E_TEMPORAL=1` для continue-as-new и bounded-retry.
- **Зависимости**: без новых (всё на установленном `temporalio>=1.7`, фактически 1.30.0).
- **Риски**: Continue-As-New требует аккуратной передачи состояния и сигналов во время сброса; retry-политики меняют поведение при сбоях (раньше — бесконечный retry, теперь — ограниченный, что может приводить к падению воркфлоу при длительной недоступности БД/LLM — это сознательный trade-off).
