# Proposal: Hotel Concierge Agent (add-concierge-agent)

## Why

Забронировать отель — просто; а вот «пост-бронирование» — ранний заезд, апгрейд номера, особые
пожелания — это нудная email-переписка с отелем, часто на чужом языке и в его рабочие часы.
Клиенты откладывают или бросают это и теряют ценность брони. kkr-hotel-assist закрывает эту
боль: клиент просто пересылает подтверждение — а интеллектуальный агент на LangGraph ведёт
переговоры с отелем по email за клиента, долговременно (днями), на языке отеля, и возвращает
готовый результат. Это первый шаг продукта; делать сейчас, потому что вся	value для пользователя
лежит именно в пост-бронировании, а не в поиске/букинге (перегруженном рынке).

## What Changes

Greenfield — новой системы пока нет. Добавляется:

- **Email-консьерж**: клиент пересылает confirmation (+ свободные пожелания) на персональный
  ящик `c.<token>@kkr-hotel.com` → агент разбирает письмо, определяет отель, его контакт и язык.
- **Агент на LangGraph** извлекает данные из писем, формирует текст и ведёт интеллектуальную
  многотурновую переписку с отелем; имеет тул `web_search` (уточнить детали отеля / найти контакт).
- **Durable-оркестрация на Temporal**: переписка живёт днями, корректно переживает ожидание,
  таймауты, фоллоу-апы, ретраи, рестарты.
- **Пакетные переговоры**: дефолтные темы (ранний заезд, условия апгрейда) + пожелания
  укладываются в одно письмо отелю (на языке отеля, fallback английский); итог — отчёт клиенту.
- **Долгоживущий контекст брони**: после отчёта клиент может дать новый ввод (по email на v1) —
  агент удерживает контекст и повторно идёт к отелю.
- **Плагабельная канальная архитектура**: email/Mailgun на v1, с архитектурной готовностью к
  омиканальности на клиентской стороне (Telegram / WhatsApp / нативное приложение с агентом) и к
  замене почтового провайдера — через адаптеры.

Явные **Non-Goals**: платежи и cost-binding решения агентом (стоимость апгрейда — только инфо для
клиента); реализация не-email каналов (только готовность портов); интеграции с API booking-платформ;
полная автономия на необратимых действиях.

## Capabilities

### New Capabilities

- `booking-intake`: приём пересланного подтверждения (+ пожеланий) на `c.<token>@`, разрешение
  клиента, извлечение структурированных данных брони и списка тем (check-in, upgrade, пожелания),
  определение отеля и его контакта/языка (если есть в подтверждении).
- `hotel-negotiation`: агент-управляемый разговор с отелем — пакетная отправка email на языке
  отеля, ожидание/разбор ответов, фоллоу-апы по таймауту/неоднозначности, разрешение тем,
  обнаружение контакта+языка через веб при отсутствии, формирование отчёта.
- `client-communication`: клиентская сторона разговора — доставка отчёта/уведомлений и приём
  фоллоу-апа, **независимо от канала** (email на v1; готовность к Telegram/WhatsApp/native).
- `messaging-gateway`: транспортная абстракция каналов — порт-пара (inbound-normalizer +
  outbound-gateway) с идемпотентной отправкой, верификацией подписи и catch-all маршрутизацией;
  реализация email/Mailgun на v1, другие каналы/провайдеры подключаются как адаптеры.

### Modified Capabilities

Нет — проект greenfield, существующих спецификаций в `openspec/specs/` пока нет.

## Impact

- **Код**: новый кодбейз по Clean Architecture (`src/domain`, `src/infrastructure`, `src/presentation`); воркфлоу/активити Temporal, LangGraph-графы, FastAPI-эндпоинты (вебхуки + клиентский API).
- **Инфраструктура (новая)**: Temporal server; PostgreSQL (домен + отдельная схема под LangGraph PostgresSaver); аккаунт Mailgun с catch-all route и подписанным вебхуком; DNS/MX на `*@kkr-hotel.com`.
- **Зависимости**: temporal (python SDK), langgraph, mailgun (HTTP/API), sqlalchemy, fastapi, uv; runtime — `uv`.
- **Внешняя зависимость**: Mailgun (inbound webhook, outbound sending API) — за адаптером, заменяем.
- **Зависимость вне скоупа**: онбординг клиента / выдача `c.<token>` (как клиент получает свой адрес) — не покрывается этим изменением, нужен до/параллельно.
- **Безопасность/PII**: пересылаемые подтверждения содержат персональные данные; требования к retention/удалению и at-rest хранению уточнить (Open Question в design.md).
- **Service-facing**: ни одно существующее API не меняется (greenfield); добавляются новые вебхук-эндпоинты.
