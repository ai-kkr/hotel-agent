## Context

Inbound-письма попадают в систему через нормализатор Mailgun-вебхука. Текущая реализация
(`src/infrastructure/mail/mailgun.py::MailgunWebhookNormalizer.parse`) читает тело как
`_first(payload, "body-plain", "stripped-text")` — только plaintext-поля Mailgun. Аналогично
поступает stub-адаптер (`src/infrastructure/mail/stub.py`). Ни один из них не смотрит на
`body-html`/`stripped-html`. Для писем без `text/plain`-части (HTML-only подтверждения) тело
оказывается пустым → экстрактор `ConfirmationExtractor` получает пустой `forwarded_payload` и не
может извлечь данные брони (по спеке `booking-intake` → low-confidence → запрос уточнения).
`html2text` уже используется в dev-утилите `scripts/` для той же задачи.

## Goals / Non-Goals

**Goals:**
- HTML-only подтверждения корректно доходят до экстрактора в виде читаемого текста.
- Plaintext-письма продолжают работать без изменений (plaintext предпочтительнее HTML).
- Логика HTML→text — в одном месте, без дублирования между Mailgun- и stub-адаптерами.

**Non-Goals:**
- Не меняем доменную модель `InboundEmail`/`ConfirmForward` и сам экстрактор (LLM получает тот же
  payload, только непустой).
- Не кормим LLM сырым HTML намеренно (выбран путь «текст предпочтительнее» — см. D2); это можно
  пересмотреть отдельным change'ом, если_extraction на сконвертированном тексте окажется хуже.
- Не добавляем replay-механизм для `.eml` — это scope скриптового change'а.

## Decisions

### D1. Fallback body-plain → html2text(body-html), не «кормить LLM сырым HTML»
**Решение:** нормализатор предпочтительно берёт `body-plain`/`stripped-text`; если пусто — берёт
`body-html`/`stripped-html` и прогоняет через `html2text` в plaintext.

**Почему не альтернативы:**
- *Кормить LLM сырым HTML* — современные модели извлекают и из HTML, но это больше токенов, хуже
  детерминированность, и ломает консистентность: для multipart-писем LLM уже получает plaintext, а
  для HTML-only — вдруг HTML. Единый plaintext-контракт проще и стабильнее.
- *Оставить как есть и формализовать «требуется text/plain»* — теряем реальные HTML-only брони.

**Порядок источника:** `body-plain` → `stripped-text` → `html2text(body-html)` →
`html2text(stripped-html)`. `stripped-*` предпочтительнее `body-*`, когда оба есть, т.к. без
quoted-части (уже сейчас `stripped-text` стоит вторым в fallback-цепочке; сохраняем симметрию).

### D2. Единая утилита в src/infrastructure/mail/
**Решение:** новый модуль `src/infrastructure/mail/html.py` с функцией `html_to_text(html: str) ->
str` (обёртка над `html2text` с фиксированными настройками: без переноса строк, без картинок).
Mailgun- и stub-нормализаторы вызывают её. Это исключает дублирование и даёт одну точку для
тонкой настройки (кодировка, нормализация пробелов и т.п.).

**Почему не в `domain/`:** HTML→text — инфраструктурная забота (формат внешнего провайдера), домен
работает уже с чистым `InboundEmail.body`. Clean Architecture: зависимость направлена
infrastructure → утилита, домен не зависит.

### D3. html2text — основная зависимость
**Решение:** перенести `html2text` из `[project.optional-dependencies].dev` в
`[project].dependencies`. Продакшен-код теперь от неё зависит.

### D4. Где применять конвертацию — на уровне нормализатора
**Решение:** конвертация происходит в нормализаторе при формировании `InboundEmail.body`, до
домена. Домен и экстрактор остаются неизменными и не знают про HTML.

**Почему не в экстракторе:** экстрактор — порт/интерфейс, не должен знать форматы провайдера; плюс
stub/webhook должны отдавать домену одинаково подготовленные данные.

## Risks / Trade-offs

- **[html2text может искажать таблицы/колонки подтверждений]** → настройки конвертера подобраны
  (`body_width=0`, без картинок); при потерях качества — точка для тюнинга одна (`html_to_text`).
  Покрыть тестом реалистичный HTML-фрагмент (таблица с данными брони).
- **[Mailgun иногда отдаёт оба поля или `body-plain` как автогенерат из HTML]** → `body-plain`
  остаётся предпочтительным; fallback срабатывает только когда plaintext пуст. Автогенерат Mailgun
  нас устраивает (это уже текст).
- **[Рост token-расхода незначителен]** → сконвертированный текст короче HTML; общий расход не
  растёт, скорее падает относительно «кормить HTML».
- **[Обрыв на некорректном HTML]** → `html2text` толерантен к битому HTML; оборачивать в
  best-effort (не ронять inbound из-за конвертации — fallback на snippet/raw-HTML при ошибке).
- **[Stub- и Mailgun-пути должны оставаться симметричными]** → оба вызывают общую `html_to_text`;
  тесты покрывают оба адаптера.
