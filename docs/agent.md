# Агент

Поведенческая спецификация LLM-агента: его роль, инструменты, промпты и правила. Структурное
размещение кода и потоки данных описаны в [architecture.md](architecture.md).

## Роль

**Персональный консьерж по поездке.** Агент доступен гостю с первого сообщения и ведёт весь
сценарий сам: разбирает пересланное подтверждение брони, уточняет пожелания, ищет контактный
email отеля, составляет и отправляет письмо на нужном языке, autonomously ведёт переписку с
отелем и информирует гостя о прогрессе. Вовлекать гостя нужно минимально — только когда без его
ответа переписку продолжить нельзя.

Два финала задачи:

- **Успех** — собрать броню через `set_booking_info` и отправить письмо `send_wishes_to_hotel`,
  либо просто ответить гостю, если вопрос решается без письма в отель.
- **Отмена** — `cancel_task` с понятной причиной, если продолжать невозможно (например, не
  удалось найти email отеля).

Агент реализован одним ReAct-графом через [`langchain.agents.create_agent`](../src/agent/agent.py)
(узлы `model` + `tools`, state-схема `EmailState`, контекст `EmailContext`, middleware
`SelfCorrectionMiddleware`). История переписки привязана к клиенту через
`ClientORM.thread_id` (`client:{id:04d}`).

## Языковая политика

- **С гостем** — исключительно на русском. Имена, названия отелей и т. п. остаются как есть, но
  весь текст и пояснения — по-русски.
- **С отелем** — на языке отеля (`hotel_language`):
  - `ru` — отель в России,
  - `zh` — отель в Китае (упрощённые иероглифы),
  - `en` — любая другая страна.
  Язык определяет сам агент по стране/названию отеля и фиксирует через `set_booking_info`.

## Инструменты

Все инструменты — в [`src/agent/tools/`](../src/agent/tools/). Тулы, меняющие state, возвращают
`Command(update=...)` (а не `dict`, иначе `ToolNode` воспринял бы возвращённое как контент
`ToolMessage`, и state не обновился бы). «Тяжёлые» зависимости тулы достают лениво через
`get_context()` — runtime-контекст `EmailContext` остаётся сериализуемым.

### `set_booking_info` — [`booking.py`](../src/agent/tools/booking.py)

Записывает детали брони по мере того, как агент их узнаёт: `hotel_name`, `from_date`, `to_date`,
`hotel_email`, `guests`, `hotel_language`. Любое поле можно опустить (`None`) — оно останется
нетронутым (см. редьюсер `booking_field` в [architecture.md](architecture.md#reducer-booking_field)).
Все шесть полей должны быть заполнены до `send_wishes_to_hotel`.

### `send_wishes_to_hotel` — [`mail.py`](../src/agent/tools/mail.py)

Составляет письмо из пожеланий гостя и отправляет отелю.

- Проверяет, что вся бронь заполнена (`missing_booking_fields`); если чего-то не хватает —
  поднимает `SelfCorrectionError` (агент уточнит у гостя или дозаполнит `set_booking_info`).
- Генерирует текст письма LLM-кой по промпту `SYSTEM_LETTER_TO_HOTEL` на `hotel_language`.
- В **dev-режиме** (`settings.is_dev`) письмо уходит на email самого гостя (`user_email`),
  а не отелю — чтобы проверять отправку без реального отеля.
- `From` = проверенный отправный адрес (`mailtrap_from_email`), `Reply-To` = inbox гостя — чтобы
  ответ отеля вернулся в его inbound-ящик и прошёл через webhook.
- Сохраняет `OutboundEmailORM` (с `message_id`) — для последующего сопоставления ответа отеля по
  `In-Reply-To`. `message_id` кладётся в state (`last_outbound_message_id`).

### `reply_to_hotel` — [`mail.py`](../src/agent/tools/mail.py)

Продолжение переписки с отелем в той же ветке. `message` — готовый текст ответа, который пишет
**сам агент** (на языке отеля, plain text, без markdown и без темы). Тул выставляет
`In-Reply-To`/`References` = последний `message_id` отеля и тему `Re: <subject>`. Доступен только
после того, как пришёл ответ отеля (иначе — `SelfCorrectionError`).

### `inform_step` / `cancel_task` — [`narration.py`](../src/agent/tools/narration.py)

- `inform_step(step)` — короткое сообщение гостю о значимом этапе («Ищу контактный email…»,
  «Отправляю письмо…»). Без злоупотребления: одно сообщение на этап.
- `cancel_task(reason)` — отменяет задачу, флаг `task_cancelled = True`, причина доносится до гостя.

### `search_internet` / `extract_web_page` — [`search.py`](../src/agent/tools/search.py)

Веб-поиск через Tavily — для поиска email отеля, официального сайта, ответов на вопросы гостя
(меню, расположение, достопримечательности). `extract_web_page` достаёт читаемый текст страницы
(например, контактной). Счётчик `search_rounds` ведётся в state.

## Промпты — [`src/agent/prompts/`](../src/agent/prompts/)

- **`system_main.md`** — системный промпт агента («концьерж»). Задаёт главную цель, правила
  поведения (запрос брони до действий, работа с пересланными письмами, ответы отеля,
  самокоррекция), дисциплину (общение по-русски, не выдумывать данные, каждый ход — ровно одна
  тула или уточняющий вопрос текстом) и правила лёгкого Markdown для Telegram.
- **`system_letter_to_hotel.md`** — отдельный промпт для генерации письма отелю (используется
  только внутри `send_wishes_to_hotel`). Определяет структуру письма, языковые правила и
  «дисциплину вывода»: только тело письма, без темы, без markdown, без выдуманных фактов.

## Самокоррекция

Тулы поднимают `SelfCorrectionError`, когда агент нарушил предусловие (отправляет письмо без
полной брони, отвечает отелю, не получив его письма). [`SelfCorrectionMiddleware`](../src/agent/middleware.py)
перехватывает такое исключение и превращает его в `ToolMessage`-подсказку — агент видит её на
следующем ходу и исправляется, а не роняет граф.

## Стриминг ответов — `stream_graph`

[`src/agent/stream.py`](../src/agent/stream.py) на один ход агента:

- определяет, не стоит ли граф на `interrupt` (тогда возобновляет `Command(resume=msg)`), иначе
  начинает новый ход;
- всё время работы держит статус «печатает…» (через `asynccontextmanager`, пере-триггерясь каждые
  ~4 с);
- стримит `custom`-сообщения (narration тулов) сразу в чат;
- накапливает чанки `AIMessage` по `message.id` и отправляет каждое итоговое сообщение целиком;
- перед отправкой конвертирует Markdown → Telegram entities и подставляет `$user_inbox`.
